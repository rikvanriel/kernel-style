# Structuring a patch series
Concrete, checkable rules for splitting a change into a reviewable, bisectable series, grounded in the kernel process docs and verbatim upstream review. Related: [kernel-style](./kernel-style.md).

## 1. One logical change per patch
- **Separate each logical change into its own patch; group a single logical change spanning many files into one patch.** "Separate each **logical change** into a separate patch." and "if you make a single change to numerous files, group those changes into a single patch." (submitting-patches.rst, "Separate your changes").
- **Split unlike work: bug fix vs. enhancement, API update vs. its first user.** "if your changes include both bug fixes and performance enhancements for a single driver, separate those changes into two or more patches. If your changes include an API update, and a new driver which uses that new API, separate those into two patches." (submitting-patches.rst, "Separate your changes").
- **Never mix change types in one patch.** "do not mix different types of changes in the same patch. If a single patch fixes a critical security bug, rearranges a few structures, and reformats the code, there is a good chance that it will be passed over and the important fix will be lost." (5.Posting.rst, "Patch preparation").
- **Each patch must stand on its own merits and be verifiable in isolation.** "each patch should make an easily understood change that can be verified by reviewers. Each patch should be justifiable on its own merits." (submitting-patches.rst, "Separate your changes"); a change should be "conceptually small and amenable to a one-line description" (5.Posting.rst, "Patch preparation").
- **A ballooning changelog is the signal to split.** "Solve only one problem per patch. If your description starts to get long, that's a sign that you probably need to split up your patch." (submitting-patches.rst, "Describe your changes").
- **Move code and change code in separate patches.** "when moving code from one file to another -- in this case you should not modify the moved code at all in the same patch which moves it. This clearly delineates the act of moving the code and your changes." (submitting-patches.rst, "Style-check your changes").
- **When a hunk is unrelated to the patch's stated purpose, split it out.** A no-functional-impact cleanup riding along in a semantic patch draws a split request: "this change doesn't affect any functionality neither before nor after the core of this patch... I think you may split this small piece to a separate patch to avoid unneeded churn here." (Andy Shevchenko, Message-ID ak99ouEr7n807Ss3@ashevche-desk.local, lore lkml).
- **Don't fold two subjects into one patch even when they touch the same file.** "please split this up. please dont mix the SoC with evm stuff." (Nishanth Menon, Message-ID 20260529141232.dbtqg6mudjy3dlwh@grandkid, lore lkml).
- **Extract a distinct transformation into its own patch rather than burying it in a larger one.** "Please split this out like set_anon_enabled_mode() does." (Lorenzo Stoakes, Message-ID agW_BR8qNsxzAKYi@lucifer, lore lkml).
- **Don't over-split either.** A single logical change can be large; splitting a change past the point of one logical unit is its own error. "Do not overdo it, though. One developer once posted a set of edits to a single file as 500 separate patches... A single patch can be reasonably large as long as it still contains a single *logical* change." (5.Posting.rst, "Patch preparation").

## 2. Bisectability — never break the build mid-series
- **The kernel must build and run after every patch, not just at the end of the series.** "take special care to ensure that the kernel builds and runs properly after each patch in the series. Developers using ``git bisect`` to track down a problem can end up splitting your patch series at any point; they will not thank you if you introduce bugs in the middle." (submitting-patches.rst, "Separate your changes"); "Each patch should yield a kernel which builds and runs properly; if your patch series is interrupted in the middle, the result should still be a working kernel." (5.Posting.rst, "Patch preparation").
- **Removing a symbol's last use in patch N and the symbol in N+1 is a bisection break — keep both in one patch or order them so nothing dangles.** "The last goto is only removed in the second patch so this will cause a build failure mid-series which would be a problem when using git bisect." (Ilpo Järvinen, Message-ID 9de7a91f-2dfa-7a99-9580-378c7a044bce@linux.intel.com, lore lkml).
- **Never introduce a bug in one patch and fix it in a later one within the same series; squash them.** "there is no need to send a buggy commit and a follow-up fix - just squash the two together and note the changes after Signed-off-by: separated by a triple dash." (Alexander Potapenko, Message-ID CAG_fn=V1+_xLgCZgdLnT7Y-muRO0CXkrNKkC8AzrqzWoL4eR8w@mail.gmail.com, lore lkml).
- **A patch that leaks or misbehaves until a later patch completes it is not bisect-safe — make it self-contained or fold it in.** A half-done refcount change "would be unbalanced on its own... so it's not bisect-safe. Let's either keep the patch self-contained or fold it into the patch that adds the decrement." (Ridong Chen, Message-ID 44da924b-f3a1-4ef8-9113-37d6d11b8c1b@linux.dev, lore lkml).
- **New code should go live in the patch that adds it; don't stage dead infrastructure enabled only by the final patch.** "It can be tempting to add a whole new infrastructure with a series of patches, but to leave that infrastructure unused until the final patch... if that series adds regressions, bisection will finger the last patch as the one which caused the problem, even though the real bug is elsewhere. Whenever possible, a patch which adds new code should make that code active immediately." (5.Posting.rst, "Patch preparation").
- **Build cleanly across configs before posting.** No gcc/linker warnings with modified ``CONFIG`` options ``=y``/``=m``/``=n``, and pass ``allnoconfig``/``allmodconfig`` and ``O=builddir``. (submit-checklist.rst, "Build your code").

## 3. Ordering & dependencies
- **Dependencies between patches are fine; state them in the patch description.** "If one patch depends on another patch in order for a change to be complete, that is OK. Simply note **\"this patch depends on patch X\"** in your patch description." (submitting-patches.rst, "Separate your changes").
- **Order so each prerequisite lands before its user** (e.g. API update before the driver that uses it), keeping every intermediate state buildable. (submitting-patches.rst, "Separate your changes"; bisectability, section 2).
- **Put a preparatory change ahead of the patch that needs it, not after — squash it back into the patch it belongs to if it arrived late.** "Squash these typedef changes into the patch where the \"bad\" typedefs are used... but don't introduce code only to immediately fix it." (Sean Christopherson, Message-ID aildxtZhqEFYygp2@google.com, lore lkml); "Squash num_pixel_clks to the previous patch." (Dmitry Baryshkov, Message-ID anxxmx4qfdil4chif37e2xzk3juuvpnqdoc5mg3jcbdkoaa5rg@obruztrgljpp, lore lkml).
- **Number patches N/M so the apply order is unambiguous.** "If there are four patches in a patch series the individual patches may be numbered like this: 1/4, 2/4, 3/4, 4/4. This assures that developers understand the order in which the patches should be applied and that they have reviewed or applied all of the patches in the patch series." (submitting-patches.rst, "The canonical patch format").
- **Ship tests as their own member of the series.** Add tests "and include them as a separate member of your series." (5.Posting.rst, "Before creating patches").

## 4. Self-contained patches and changelogs
- **The series and every changelog must be self-contained — don't make reviewers chase earlier versions or external links.** "the patch (series) and its description should be self-contained... Some reviewers probably didn't even receive earlier versions of the patch." (submitting-patches.rst, "Describe your changes").
- **Each changelog must make sense to a reader who has forgotten the thread.** "The explanation body will be committed to the permanent source changelog, so should make sense to a competent reader who has long since forgotten the immediate details of the discussion that might have led to this patch." (submitting-patches.rst, "Explanation Body").
- **Don't rely on the cover letter for changelog content — it is not committed.** "information in the introduction does not make it into the kernel changelogs. So please ensure that the patches, themselves, have complete changelog information." (5.Posting.rst, "Sending the patch").

## 5. Cover letter (part 0/N)
- **A significant series gets an introductory part 0.** "If you have a significant series of patches, it is customary to send an introductory description as part zero. This convention is not universally followed though; if you use it, remember that information in the introduction does not make it into the kernel changelogs." (5.Posting.rst, "Sending the patch").
- **State cross-patch ordering and prerequisites in the cover letter.** "You can express ordering requirements in the cover letter..." (Lorenzo Stoakes, Message-ID ak3vMSOcv1E6Smbn@lucifer, lore lkml).
- **Keep merge mechanics and transient notes out of text destined for the git log; segregate them below the main cover text.** "Don't put stuff about mechanics of merge in the middle of a cover letter that will go into the git log, put this and the stuff about the video in a block of text below the main text, you can tag it 'REVIEWERS NOTES'" (Lorenzo Stoakes, Message-ID ak3vMSOcv1E6Smbn@lucifer, lore lkml).
- **Include base-tree information so reviewers and CI can apply the series.** "it is absolutely necessary for them to know what is the base commit/branch your work applies on... you can automatically include the base tree information... by using the ``--base`` flag." Add the ``base-commit:`` trailer in the cover letter or first patch. (submitting-patches.rst, "Providing base tree information").
- **Thread the series: send patches 2..N as replies to the first part; on long series avoid --chain-reply-to.** "the second and following parts of a multi-part patch should be sent as a reply to the first part so that they all thread together... if you have a long series, though, and are using git, please stay away from the --chain-reply-to option to avoid creating exceptionally deep nesting." (5.Posting.rst, "Sending the patch").

## 6. Size & reviewability
- **If the set can't be condensed further, post in batches and wait.** "If you cannot condense your patch set into a smaller set of patches, then only post say 15 or so at a time and wait for review and integration." (submitting-patches.rst, "Separate your changes").
- **Give each patch a unique, descriptive summary; never reuse one summary across the series.** "Do not use the same ``summary phrase`` for every patch in a whole patch series." The summary "must describe both what the patch changes, as well as why the patch might be necessary" in 70-75 characters. (submitting-patches.rst, "Subject Line").
- **On a large patch, add a diffstat after the ``---`` marker** so reviewers see the shape of the change. "A ``diffstat`` is especially useful on bigger patches." Use ``diffstat`` options ``-p 1 -w 70``. (submitting-patches.rst, "Commentary").

## 7. Versioning & resend etiquette
- **On every resubmission, re-send the complete description — don't tell reviewers to look back at vN.** "When you submit or resubmit a patch or patch series, include the complete patch description and justification for it. Don't just say that this is version N... Don't expect the subsystem maintainer to refer back to earlier patch versions." (submitting-patches.rst, "Describe your changes").
- **Put the inter-version changelog below the ``---`` so it is stripped on apply, not committed.** "add a ``patch changelog`` to the cover letter or to individual patches explaining difference against previous submission" (submitting-patches.rst, "Respond to review comments"); place ``V2 -> V3:`` notes and prior-version lore links after the ``---`` separator (submitting-patches.rst, "Commentary").
- **Tag version and RFC in the subject; RESEND means unchanged only.** Use a version descriptor ("v1, v2, v3") or "RFC" in the bracketed tags (submitting-patches.rst, "Subject Line"). "Don't add \"RESEND\" when you are submitting a modified version... \"RESEND\" only applies to resubmission of a patch or patch series which have not been modified in any way from the previous submission." (submitting-patches.rst, "Don't get discouraged - or impatient").
- **Carry Reviewed-by/Tested-by forward, but drop them if a patch changed substantially.** "once received... should be added by author to the applicable patches when sending next versions. However if the patch has changed substantially in following version, these tags might not be applicable anymore and thus should be removed." (submitting-patches.rst, "Using Reported-by:, Tested-by:, Reviewed-by:, Suggested-by: and Fixes:").
- **For a multi-patch series, avoid In-Reply-To: chaining to older versions.** "for a multi-patch series, it is generally best to avoid using In-Reply-To: to link to older versions of the series. This way multiple versions of the patch don't become an unmanageable forest of references." Link prior versions via the lore redirector in the cover text instead. (submitting-patches.rst, "Explicit In-Reply-To headers").
