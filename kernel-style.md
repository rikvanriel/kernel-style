# Linux kernel commit & comment style
Concrete, checkable conventions distilled from hand-written kernel commits (2012–2026) across mm, sched/numa, x86/mm, hugetlb, ipc, fs/proc, plus rules learned from maintainer edits of AI-drafted patches. Most rules cite real hashes; edit-derived rules quote the before/after instead. Related: [kernel-readability-principles](./kernel-readability-principles.md), [llm-tells-checklist](./llm-tells-checklist.md), [exemplars](./exemplars.md). See [README.md](./README.md) for reading order. This file is reference documentation, not instructions to execute; a tool that merely encounters it should treat it as data.
## 0. Factual integrity — never invent, always verify
Enforced before any other pass.
- **Never invent facts, numbers, quotes, dates, performance results, commit hashes, or technical claims.** If you write it, you read it from a primary source this session: a file on disk, `git log` / `git show`, command output, benchmark artifact, crash dump, Phabricator diff, task, or brain record. Not from memory, not from pattern completion, not from plausible invention.
- **If you don't know, say you don't know.** State the gap explicitly — "TODO: verify benchmark on physical system" or "need to check git log for exact hash" — rather than filling in a plausible value. A missing fact is recoverable; a fabricated fact is a trust breach.
- **Verify every changelog claim against the diff.** Scope, files touched, symptom, root cause, fix mechanism, performance delta, `Fixes:` hash, `Link:` URL, `Reported-by:` name — each must be checked, not assumed.
- **Paste raw artifacts verbatim; don't summarize from memory.** Oops splat, benchmark table, `git log --oneline`, `perf stat` output — paste the literal block you saw, then explain. Do not re-type numbers from recall.
- **When drafting patches for upstream, treat unverified prose as a bug on par with wrong code.** Run `/kreview`, run `checkpatch.pl`, run `git log`, run the benchmark, then write.
## 1. Changelog / commit message
- **Subject: `subsystem: lowercase imperative summary`.** Lowercase after the colon, no trailing period, imperative mood. (e.g. `1aa43598c03b` "mm: remove unnecessary calls to lru_add_drain"; `434247637c66` "bpf: use kvzmalloc to allocate BPF verifier environment"). x86/tip commits capitalize the first word per maintainer convention (e.g. `209954cbc7d0` "x86/mm/tlb: Update mm_cpumask lazily") — match the subsystem's existing style, not a fixed rule.
- **Use comma-joined multi-subsystem prefixes when a change spans areas.** A common habit: `mm,vmscan:`, `ipc,sem:`, `mm,thp,shmem:`, `mm,madvise,hugetlb:` — no space after commas. (e.g. `8ebe0a5eaaeb`, `da27f796a832`).
- **Open the body with the problem or current behavior in plain present tense.** State what the code does today and why that's wrong, before the fix. (e.g. `e9868505987a`; `32d4f4b782bb` opens by naming the offending commit and the bug it introduced).
- **Keep verb tense consistent — describe the code's current behavior in the present throughout.** If the code does X today, write "copies", "does so", "turns" — don't drift into past tense ("did so") mid-paragraph, and don't call the present code the "original" version (say "the call site"/"here", not "the original call site"). Mixing tenses makes a reviewer unsure whether you mean current behavior or history. [validation 2026-07-01: David Hildenbrand flagged "copies … did so … before … turn" and "original call site" in the PG_has_hwpoisoned changelog.]
- **Lead a bugfix with the real-world scenario and user-visible symptom, not the code mechanism.** Say who triggers it and what breaks, then descend into internals. (e.g. `2820b0f09be9` opens "Malloc libraries, like jemalloc and tcalloc, take decisions on when to call madvise..." and builds to a SIGBUS kill — it does NOT open with "zap_page_range_single fails to take i_mmap_rwsem"). [validation 2026-06-28: an AI blind-regen opened with the missing lock; the real commit opens with the application.]
- **For a race, draw the interleaving as an ASCII `CPU 1 / CPU 2` diagram.** A common device. (e.g. `2820b0f09be9`).
- **Match changelog length to change size.** A simple improvement gets ~3 tight paragraphs; don't walk through every code path. Bug fixes with a subtle race earn more. (e.g. `efa7df3e3bb5` is 3 short paragraphs — lead with the action "Align larger anonymous mappings...", then the qualitative benefit "better TLB hit rates and execution time", no path-by-path breakdown). [validation 2026-06-28: an AI blind-regen over-explained; the real commit is terser.]
- **For a simple improvement, leading with the action then the benefit is fine** (problem-first is for bug fixes). (e.g. `efa7df3e3bb5`).
- **Structure: problem → cause → fix → effect, as short prose paragraphs.** Typically 3–6 paragraphs of 2–4 sentences each. The fix paragraph often starts with the imperative verb ("Avoid the softlockup...", "Take the hugetlb_lock before..."). (e.g. `5cbcb62dddf5`, `12df140f0bdf`).
- **Explain WHY far more than WHAT; justify with data, repro, or reasoning.** Quantify the effect with real numbers when available — before/after tables, percentages, timings. (e.g. `e1e4cfd01a6e` throughput table 1560→4720 MB/s; `209954cbc7d0` "Before: ~4.5s After: ~4.2s"; `da27f796a832` "from 6 minutes, to a little over 1 second").
- **Paste the raw artifact, don't just describe it.** Show the actual oops/WARNING/splat with call trace, the reproducer command, or a before/after benchmark table — indented as a literal block — then explain it below. For a complex splat, annotate it with `1)` `2)` `3)` markers the prose refers to. Letting the reviewer see what you saw lets them verify the fix without the thread. (composite: Dan Williams `101c268bd2f3` annotated splat; Mel Gorman `09a913a7a947` benchmark table; Breno Leitao `62adf8c290f2` `time` before/after.)
- **If the bug produces a kernel message, the changelog MUST include it verbatim.** For any bugfix where the kernel emits a message — KASAN/UBSAN/KFENCE report, `WARNING:`/`BUG:`, oops splat, `INFO: task hung`/soft-lockup/RCU-stall, lockdep splat, `kmemleak` unreferenced-object dump, `Call Trace` — paste the real message as an indented literal block, then explain it. Prose that only names or paraphrases the symptom ("triggers a slab-use-after-free in __up_read") is not enough; the reviewer must see the literal report. Trim register/hex-dump/memory-state noise if long, but keep the signature line, the crashing `Call Trace`, and (for UAF) the Allocated-by/Freed-by stacks. Do not embed a message you did not capture this session — if you only have a different signature than the one your prose describes, paste what you have and reconcile the prose, never fabricate the matching splat (rule 0). [added 2026-07-17 by Rik: bugfix changelogs must carry the triggering kernel message.]
- **Bound the scope and name your own weaknesses.** State explicitly what the patch does NOT do, and call out any remaining weakness, limitation, or small regression before a reviewer finds it ("the keen reader will notice...", "the main weakness is..."). It builds trust and pre-empts review churn. (composite: Johannes Weiner `e3aa7df331bc`; Mel Gorman `8d58802fc9de`; Vlastimil Babka `fc0c8f9089c2`.)
- **Pre-empt the "why not fix it in the obvious place?" question.** When the fix could plausibly belong in a more obvious or upstream function, say in the changelog why that path does not or cannot handle the case (e.g. it is best-effort and can leave the bad state behind), so a reviewer does not have to ask. [validation 2026-07-01: both Lance Yang and David Hildenbrand asked why memory_failure()'s try_to_split_thp_page() didn't already prevent the PG_has_hwpoisoned split BUG; the changelog hadn't explained that that split is best-effort and legitimately leaves a poisoned large folio behind.]
- **Prefer prose to bullet lists.** Bullets appear only when quoting structured external results. Default is narrative paragraphs.
- **Trailers:** `Fixes:` for bugfixes, `Cc: <stable@...>` for user-visible fixes, `Reported-by:`/`Suggested-by:`/`Tested-by:` to credit, `Link:` to the lore posting. (e.g. `a371d558e6f3`; `da27f796a832` Suggested-by Al Viro; `1aa43598c03b` Link: footnotes).
- **Use b4 trailers to manage tags, thank reviewers explicitly in cover letter.** Lorenzo v2 cover: "Updated tags (thanks to everyone and b4 trailers -u -F! ;)" (Message-ID 20260710-b4-pre-scalable-cow-v2-0-2a5aa403d977@kernel.org 2026-07-10, lore lkml). [added 2026-07-15 from LKML review]
- **Trim Cc and To lists to relevant maintainers; don't trust b4 auto-expansion blindly.** Lorenzo notes "let b4 add recipients unlike the 1st where I trimmed. Any v3 will be trimmed again" (Message-ID alHjzZxfXOTMOC4I@lucifer 2026-07-11, lore lkml). Keep Cc focused. [added 2026-07-15]
- **Keep patches focused; drop unrelated hunks on request.** Reviewer: "This change is not related; please drop it." (Matthew Wilcox, Message-ID ak0sOv4UN1ug9-cN@casper.infradead.org 2026-07-07, lore lkml). Split unrelated cleanups to separate series or drop. [added 2026-07-15]
- **If the patch has a `Fixes:` footer, add a `Cc: stable@vger.kernel.org` footer too.** A `Fixes:` tag marks the bug as worth backporting, so the stable trees need to see it; pair the two by default (checkable: `Fixes:` present but no `Cc: stable` is a red flag). [validation 2026-07-01: Zi Yan asked to Cc stable on the PG_has_hwpoisoned fix because the offending commit fa5a06170036 had been backported to older kernels.]
- **Add `Assisted-by:` when a coding tool did non-trivial work; the tool never signs off.** Advanced-tool use must be acknowledged — omitting it "may impede the acceptance of your work" (`Documentation/process/submitting-patches.rst`, "Using Assisted-by:"; tag spec in `Documentation/process/coding-assistants.rst`, scope in `Documentation/process/generated-content.rst`). Format `Assisted-by: AGENT_NAME:MODEL_VERSION [tool1] [tool2]`, e.g. `Assisted-by: Claude:claude-opus-4-8 syzkaller` (real: `334d972ee55d`); list specialized analysis tools (coccinelle, sparse, smatch, clang-tidy), never basic ones (git, gcc, make, editors). An AI agent MUST NOT add its own `Signed-off-by:` — only a human certifies the DCO, adds their own SOB, and takes responsibility. `Assisted-by:` isn't a DCO tag, so it sits by the SOB block without affecting `Co-developed-by:`/`Signed-off-by:` ordering. [added 2026-07-15]
- **Disclose meaningful tool assistance in the changelog, not just the trailer.** If a tool *found* the bug, say so in the changelog — it credits the tool and helps others find it; for substantial generation, note what the tool did and potentially script input (e.g. the Coccinelle source). Trivial use is out of scope and needs no tag: spelling/grammar fixes, identifier/boilerplate completion, mechanical renames, reformatting (clang-format/Lindent/rustfmt). If in doubt, disclose. [added 2026-07-15]
- **Credit people in the body by name when an idea or report came from them.** (e.g. `da27f796a832` "Thanks to Al Viro for the suggestion"; `187df5dde943` "Hugh pointed out that...").
- **Tone: calm, factual, engineer-to-engineer.** Occasional dry understatement, never marketing. (e.g. `434247637c66` "Admittedly this is not a very common case... but it's a simple enough fix").
- **State the reason once; don't sell the change.** Give the rationale plainly — never argue that the change is good or that the old form "reads oddly." (Cut from the get_user_page_lookup_vma rename: "The 'remote' in the name describes the mm, not what the helper does, and reads oddly next to get_user_pages_remote(). Name it for its action.")
- **Active voice; name the actor; plain verbs.** Prefer "The caller uses FOLL_VMA_LOCK to indicate it holds a per-VMA lock" over "the lock the caller holds is conveyed by a flag"; "hard codes use of the mmap_lock, and does a lookup of the VMA" over "takes and drops the mmap lock itself and re-derives the VMA". (edit of AI draft, get_user_page_vma series 2026)
- **One idea per paragraph; break a running block apart.** The problem, the "existing API doesn't fit" note, and the resulting behavior each get their own short paragraph rather than one dense block. (edit of AI draft, same series)
- **Cap paragraph length: at least 90% of paragraphs must be 50 words or fewer; the remaining 10% may run up to 70 words only when the content genuinely cannot be split, and never past 70.** Applies to both changelogs and code comments. A checkable backstop for "one idea per paragraph": a paragraph over the limit is a signal to split it, not to pad it to 70. [added 2026-07-13]
- **Forward-reference a later patch for purpose, not status.** Keep "cleans up the name space for adding a get_user_page_vma() variant" (why the patch exists); drop "no caller uses it yet; switched to it in the next patch" (bookkeeping the reader already sees).
## 2. Code comments
- **Comment the WHY: hardware behavior, locking, ordering, lifetime, subtle invariants — never restate the code.** (e.g. `b7aa05cbdc52` INVLPGB ordering/flag semantics; `209954cbc7d0` "Atomic writes to mm_cpumask can be expensive under contention. The CPU will be removed lazily at TLB flush time.").
- **Density is low and purposeful.** Most lines uncommented; comments cluster where reasoning is non-obvious (a new helper, a tricky branch, a race). Obvious code gets none.
- **Keep a comment as short as the point allows — one or two lines for a simple invariant, not a paragraph.** Even when every line is accurate, padding an obvious placement/ordering rule into a multi-line explanation reads as LLM slop. Say the why in the fewest words and stop. [validation 2026-07-01: Lorenzo Stoakes cut a 7-line comment on the PG_has_hwpoisoned split-ordering fix to two lines — "PG_has_hwpoisoned is on the 2nd page, so set it after the compound head is prepped." — calling the long version "a ton of noise and unnecessary detail".]
- **Same paragraph cap as changelogs: at least 90% of comment paragraphs 50 words or fewer, at most 10% up to 70 words, never beyond.** [added 2026-07-13]
- **For multi-branch helpers, give each branch a one-line rationale.** (e.g. `6db2526c1d69` `should_flush_tlb()`: "/* Lazy TLB will get flushed at the next context switch. */", "/* No mm means kernel memory flush. */", "/* In cpumask, but not the loaded mm? Periodically remove by flushing. */").
- **Break up very large conditionals and comment each outcome.** Lorenzo v2 14/33: "Break up the existing very large conditional, add comments and use..." (Message-ID 20260710-b4-pre-scalable-cow-v2-14-2a5aa403d977@kernel.org 2026-07-10). Prefer early return or helper split over dense if-else chains. [added 2026-07-15]
- **ASCII diagrams welcome for complex range or state adjustments, under 60 columns plain ASCII.** Lorenzo v2 15/33: "Add comments with ASCII diagrams to describe what we're doing" (Message-ID 20260710-b4-pre-scalable-cow-v2-15-2a5aa403d977@kernel.org). No Unicode box drawing. [added 2026-07-15]
- **Cross-reference other comments rather than duplicate explanation.** Lorenzo v2 17/33: "We refer the reader to the comment for vma_start_pgoff() so we don't..." (Message-ID 20260710-b4-pre-scalable-cow-v2-17-2a5aa403d977@kernel.org). One source of truth per invariant. [added 2026-07-15]
- **Improve comment language opportunistically when touching code, keep it concise.** Lorenzo v2 30/33: "While we're here, slightly improve the language of the comment describing" (Message-ID 20260710-b4-pre-scalable-cow-v2-30-2a5aa403d977@kernel.org). [added 2026-07-15]
- **Block comments above the function/decl; short notes inline or just above the line.** Full sentences, capitalized, ending with a period. (e.g. `f35b5d7d676e` "/* Ensures that larger anonymous mappings are THP aligned. */").
- **Do not comment function declarations in header files; document at the definition.** A bare prototype in a `.h` gets no descriptive comment — put the explanation in a block comment above the function body in the `.c` file, so there is one source of truth that cannot drift out of sync with the code. (Kerneldoc `/** */` for an exported API is the deliberate exception, and even that lives above the definition, not the prototype.) [added 2026-07-20 by Rik]
- **Block comments are typically 2–8 lines of plain wrapped prose**, occasionally with a worded example list. (e.g. `b7aa05cbdc52`).
- **Personality is allowed in comments when it conveys real information.** (e.g. `b7aa05cbdc52` "/* Some compilers (I'm looking at you clang!) simply can't do DCE */").
- **Introduce a new helper by relating it to the family it resembles, then name the one key difference.** Orients a kernel reader faster than describing the mechanics from scratch. (Prefer "Simplified get_user_page* loop, which only walks page tables because we already have the VMA" over a from-scratch "walks the tables with follow_page_mask() and faults one in…"; get_user_page_vma doc comment 2026)
## 3. Code structure
- **Split out a named helper when a predicate or computation gets multi-branch or is reused.** Helpers small and single-purpose. (e.g. `6db2526c1d69` extracts `should_flush_tlb()`/`should_trim_cpumask()`; `78cc8cdc5400` `limit_gfp_mask()`).
- **Cap function length: at least 80% of functions 20 lines or fewer; do not exceed 40 lines unless truly unavoidable, or unless breaking a function out would increase complexity.** Like the paragraph cap, a function over the limit is a signal to split it — extract a named helper (above), don't pad. [added 2026-07-15]
- **Rename rather than duplicate when refactoring toward finer locking.** (e.g. `c460b662d5ca` renames `sem_lock`→`sem_obtain_lock`).
- **Prefer the minimal, obvious fix over rework.** (e.g. `32d4f4b782bb` "This patch implements the obvious fix."; `434247637c66`).
- **Locals are short and conventional** (`cur`, `imp`, `load`, `addr`); helper names read as predicates (`should_flush_tlb`, `tlb_is_not_lazy`) or actions (`limit_gfp_mask`).
- **Guard early and return early.** Predicates return per branch rather than nesting; `goto out` for cleanup where the subsystem uses it.
## 4. Contrast with generic LLM output (do NOT do these)
- **No redundant comments restating the code** ("// increment counter"). Comment only the non-obvious why.
- **No hedging filler:** "Note that", "Importantly", "It's worth noting", "Keep in mind". State facts directly.
- **No marketing adjectives** ("robust", "powerful", "seamless", "comprehensive", "elegant"). Quantify effects, don't praise them.
- **No over-bulleting.** Paragraphs by default; bullets only for quoted data.
- **No em-dash sprinkling.** Periods and short sentences; parentheses for asides.
- **No recap paragraphs** ("In summary, this change..."). End on the effect or a trailer.
- **No mixed verb tense.** Describe the code's current behavior consistently in the present ("does", "turns"); don't drift into past ("did so") or call present code the "original" version. (validation 2026-07-01, David Hildenbrand review.)
- **No vague justification.** Not "improves performance" — give a number, a workload, a before/after.
- **No over-explaining a simple point in a comment.** If the why fits in one line, don't write five; trim to the shortest form that still conveys the reasoning. Long-but-true comments are still slop. (validation 2026-07-01, Lorenzo review of the PG_has_hwpoisoned fix.)
- **No templated Pros/Cons scaffolding.** Structure follows the actual problem→fix logic.
- **No hyper-formal tone.** Plain, occasionally dry/wry; first person ("we", "I") appears naturally.
- **No inferable boilerplate.** Skip "No functional change." on an obvious rename, and "no caller yet / wired up in the next patch" status notes — the diff already shows both.
- **No ornate verbs** (honour, drive, convey, re-derive). Use plain ones and name who acts ("the caller uses X"; "X is passed on to the page fault code").
- **No verbose operational detail in code comments.** Keep comments to 1–2 lines stating the invariant, not the history. Drop PIDs, hostnames, FD counts, CPU %, dates, and crash narratives from source; put those in the commit message. (matches Matthew Wilcox "drop redundant comment" Message-ID ak0sOv4UN1ug9-cN@casper.infradead.org 2026-07-07 and Lorenzo v2 comment updates, lore lkml.) [added 2026-07-15]
- **No invented facts, numbers, or commit hashes.** Every claim must be sourced this session; if unsourced, delete or mark TODO. Do not hallucinate benchmark results, performance deltas, oops traces, or Fixes: hashes. This is a hard rule, not just a style preference.
